## Vision

### 10x Check

A verification co-pilot that makes developers feel like they have a senior QA engineer watching their back. Self-healing across runs, streaming progress, confidence scoring, timing intelligence. The pipeline learns your app and gets measurably better with every run.

### Platonic Ideal

A developer runs `/verify`, sees their ACs confirmed in under 3 minutes with screenshots, and trusts it enough to merge without manual testing. The experience feels like pair programming with someone who already knows the app. By the fifth run, the pipeline knows the app's quirks better than most developers on the team.

The feeling: "This thing just *gets* my app."

## Accepted Scope (beyond base design)

These 5 expansions are in scope for v2, in addition to the base 6-stage architecture described in the design doc.

### 1. Streaming Progress Dashboard

Live terminal view showing each AC's status as it progresses:

```
[ac1 ✓ pass] [ac2 ⏳ navigating...] [ac3 ⏳ waiting for setup] [ac4 ✗ auth redirect]
```

The user knows what's happening at every moment instead of staring at a blank terminal. During the Judge stage (60-120s), show a spinner with timing estimate: `Judging... (6 ACs, ~60s)`.

**Implementation:** New `lib/progress.ts` module. Orchestrator emits events. Each stage calls a shared progress emitter.

### 2. Confidence Scoring on Verdicts

The Judge outputs pass/fail + confidence (high/medium/low) with reasoning:
- `pass (high): screenshot shows exact banner text`
- `pass (low): text found but layout looks different than expected`

Low-confidence passes are highlighted in the report so the user knows where to look.

**Implementation:** Judge prompt change + `verdicts.json` schema update (add `confidence` field). Report formatting change.

### 3. --dry-run Mode

Run AC Generator + Planner + Plan Validator without executing browse agents. Shows:
- ACs found and how they'd be grouped
- Browser steps and URLs for each AC
- Setup SQL that would be executed

Costs one Opus + one Sonnet call (~30s) instead of a full run. Valuable for prompt iteration during eval phase and user plan validation.

**Implementation:** Orchestrator `--dry-run` flag. Stops after Plan Validator. Formats and displays plan.

### 4. Run Timing & Duration Summary

Each run reports wall-clock time, time per stage, and time per AC. The Learner tracks historical averages so future runs show: `Estimated: ~4 min based on 3 previous runs`.

**Implementation:** Report reads `timeline.jsonl`. Learner writes timing averages to learnings.md.

### 5. Self-Healing Learnings

The Learner captures *corrected SQL patterns* from failures. When setup SQL fails and the error reveals the fix (e.g., `column subscriptionStatus does not exist` → `the column is subscription_status`), the Learner writes the corrected pattern to learnings.md.

On the next run, the Setup Writer reads this and generates correct SQL without failing first. Run 2 succeeds where run 1 failed, with zero human intervention.

**Implementation:** Learner prompt reads setup errors + generated SQL, writes corrected patterns. Setup Writer prompt reads corrections from learnings.md.

## Key Architecture Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Implementation approach | Full TypeScript rewrite | Bash is structurally wrong for structured parallelism/error handling |
| User checkpoint | SKILL.md handles it (phased execution) | Natural Claude Code UX, not stdin |
| Judge streaming | Spinner, not partial verdicts | Can't stream structured output from `claude -p` |
| Teardown failures | Log warning + add to learnings | Don't block verdicts, but track for next run |
| SQL safety | Trust LLM + audit trail | Local dev tool, commands.json is inspectable |
| Learnings size | Learner prunes on every run | No hard cap, prompt says "keep concise, remove stale" |

## NOT in Scope

| Item | Rationale |
|------|-----------|
| Cloud execution | Phase 2 — requires server infrastructure |
| PR comment integration | Phase 2 — depends on cloud execution |
| Multi-ORM setup (Drizzle, TypeORM) | Ships after v2 (tracked in TODOS.md) |
| Eval infrastructure for v2 stages | Ships after v2 (tracked in TODOS.md) |
| Run comparison / --diff mode | Primarily eval-phase tool, JSON diff suffices |
| Inline terminal screenshots | Resolution too limited, HTML report sufficient |
| Parallel judge per-group | Only needed for 15+ AC specs |
